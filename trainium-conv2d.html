Optimizing a Conv2D Kernel on AWS Trainium

May 29, 2026 · ~12 min read

The target is a single AWS Trainium 1 chip: one NeuronCore-v2, with a 128×128 systolic Tensor Engine. The job is a 2D convolution written in NKI, AWS's low-level kernel language for Neuron hardware. There's a leaderboard: each test case has a latency threshold, your score is how far under it you land, and there are eleven visible shapes — a mix of float16 and float32, filters of 3×3 and 5×5, batch 4 and 16.

I ended at 3371µs total across all testcases, which was an average speedup of ~5x across the tests. Here is what I changed, in two parts: a first round of optimizations that reached a local optimum, then a weight-hoisting rewrite plus a bit of precision magic :)

Hardware constraints I focused on

Trainium isn't a GPU, and three of its constraints shaped most of the work:

  • DMA contiguity. When you nl.load a tile from HBM, every dimension except the first has to be contiguous in memory. If it isn't, the compiler doesn't error — it silently emits thousands of scalar loads and you hit a hard latency floor.
  • Eight PSUM banks. The accumulator memory has exactly eight banks per core. Ask for sixteen and you get PSUM8 — allocated memory out of bound. This limits how many matmul accumulators can be live at once, which limits pipelining.
  • A scalar processor that stalls the matrix pipeline. Integer division and modulo run on a slow scalar unit. In an inner loop they stall the Tensor Engine for microseconds at a time.

Most of what follows is hitting one of these and rearranging the kernel to avoid it.

First attempt

Load the weights once

The intuitive thing is to load each weight tile right before you use it. On Trainium that forces the input tensor X to be re-read from HBM on every weight pass. Bulk-loading all the weights into SBUF (the on-chip scratchpad) at the start means X is streamed exactly once per compute block. The early bottleneck was HBM bandwidth, not compute, so this was the biggest single lever.

The 128-loop that avoids a silent transpose

I wanted the weights laid out transposed in SBUF, and the natural move is to let a 2D DMA mapping do the transpose. The compiler accepted it, then emitted 6,400 scalar load instructions and pinned me to a ~600µs floor — the DMA engine won't treat a non-contiguous free dimension as a bulk transfer. The fix was to pull contiguous 3D strips with an explicit loop and scatter them into the stationary tiles by hand.

# contiguous 3D strips [128 channels, H, W], scattered manually
for om in nl.static_range(128):
    w_strip = nl.load(W[oct_*OC + om, ict*P + w_k, w_h, w_w])
    for kh, kw:
        weight_tiles_ci[kh][kw][k_i, om + k_j] = w_strip[k_i, kh, kw + k_j]

Ugly, but it stays on the bulk DMA path. The rewrite below replaces it.

Cache the view, slice once

For the 256-output-channel cases the kernel computes two output tiles at once. I had been slicing the moving input view twice, once per accumulator. Caching the slice into a local and feeding both accumulators halved the slicing overhead.

v0 = row_block[v_c, v_r, v_w + kw]   # slice once...
acc0 += nisa.nc_matmul(w0, v0)        # ...reuse for both OC tiles
acc1 += nisa.nc_matmul(w1, v0)

Loading the exact row block each filter row needs also makes a width shift by kw a simple pointer offset instead of a strided gather that triggers a hidden tensor_copy.

Unroll the 3×3, and stop dividing

Two instruction-level changes. For the hot 3×3 cases I deleted the kh/kw loops and wrote out nine explicit nc_matmul calls, which removes loop-control latency.

The second one is less obvious. I had flattened the batch and spatial dimensions into one 1D loop and recovered the indices with // and %. Because the NKI stack allocator forces you to skip the LLVM middle-end, those divisions ran on the scalar processor and stalled the matmul pipeline by ~10µs. Plain nested affine_range loops let the hardware's native loop controller handle the indexing instead. So the "smart" flattening was slower than the plain nested loop.

The decorator stack that ties this together is counterintuitive:

@ncc.enable_stack_allocator
@ncc.skip_middle_end_transformations
@nki.jit
def conv2d(X, W, bias, ...):
    ...

Skipping the middle-end optimizer makes sense once you notice that with manual unrolling and native loops, the AST is already the schedule you want. Letting the optimizer touch it just software-pipelined the dual accumulators into a PSUM overflow. That got me to a local optimum. To move past it I had to replace the weight-loading scheme.

The breakthrough

One HBM load instead of 128

The hand-scattered 128-loop was correct but expensive: 128 separate DMA descriptors, each scattering one strip across 128 output positions. The replacement does a single 4D block load whose last three dimensions cover the whole (ic, fh, fw) sub-cube, so contiguity holds, and then transposes each tap on the Tensor Engine.

oc_p = nl.arange(TILE_CO)[:, None, None, None]
ic_f = nl.arange(TILE_CI)[None, :, None, None]
kh_f = nl.arange(filter_height)[None, None, :, None]
kw_f = nl.arange(filter_width)[None, None, None, :]

# one contiguous 4D block -- par_dim=CO, free=(CI,FH,FW)
w_block = nl.load(W[co_t*TILE_CO + oc_p, ic_t*TILE_CI + ic_f, kh_f, kw_f])

w_buf = nl.ndarray((nl.par_dim(TILE_CI), FH, FW, TILE_CO), dtype=W.dtype, buffer=nl.sbuf)
for kk in nl.affine_range(FH * FW):    # affine, not static -- avoids unrolling all 9/25 at once
    kh_s, kw_s = kk // FW, kk % FW
    wt = nisa.nc_transpose(w_block[:, :, kh_s, kw_s])
    w_buf[:, kh_s, kw_s, :] = nisa.tensor_copy(wt)

The nine (or twenty-five) on-chip transposes cost about 0.4µs total, which is small next to the DMA serialization they replace. Using affine_range over the taps instead of nested static_range matters: it stops the compiler from unrolling every transpose at once, which had been blowing up PSUM live-ranges.

Using all eight PSUM banks

The automatic allocator was effectively using four of the eight PSUM banks. Allocating an 8-deep accumulator explicitly forces all eight into distinct physical banks.

acc_buf = nl.ndarray(
    (8, nl.par_dim(TILE_CO), ROW_BLOCK, out_width),
    dtype=nl.float32,
    buffer=ncc.psum.mod_alloc(base_bank=0, num_bank_tiles=(8,)),
)

With eight accumulators live, the next eight nc_matmuls issue back-to-back without bank stalls, and the drain of one accumulator (a vector op plus a DMA store) overlaps with the matmul feeding the next, so the pipeline doesn't bubble. Asking for sixteen banks confirmed the ceiling: NCC_INLA001 — out of bound PSUM8.

Octet X reuse, and an 18-row super-tile

When both output-channel tiles share the same input, iterating output-channel-outer loads X twice. Loading X once per group of eight blocks and letting both OC passes reuse it cuts those DMAs in half. For the 66×66 case I could also fuse the loads for two adjacent row-blocks into one 18-row super-tile. I swept 10 / 18 / 34 / 66 rows; 18 (exactly 2 × ROW_BLOCK) was best, since it overlaps each DMA with several matmuls instead of blocking on one large load.

Trading precision for speed

The last thing left was precision. The matmul cost on NeuronCore-v2 is simple:

fp32            : 4 * max(min(64, N_stat), N_moving)
fp16/bf16/tf32  :     max(min(64, N_stat), N_moving)

An fp32 matmul costs 4× a TF32 one, so demoting a tap to TF32 saves three-quarters of its cycles. The catch is correctness: the fp32 tolerance is rtol=1e-5, and TF32's 10-bit mantissa adds about 1e-3 relative error per multiply, so you don't get many taps before blowing the budget.

I treated the tolerance as a budget. A separate harness runs the kernel against PyTorch's conv2d across 30 RNG seeds from the tester's input distribution and records the worst-case ratio of error to tolerance:

diff       = np.abs(Y_kernel - Y_torch)        # per-element error
abs_budget = atol + rtol * np.abs(Y_torch)     # per-element tolerance
rel_use    = (diff / abs_budget).max()         # worst over all positions
# rel_use = 1.0 means we exactly hit the ceiling at the worst pixel

My rule: only accept anything above ~0.7 of budget for a sizeable win, and reject anything near 1.0. A single TF32 corner tap on the fp32 3×3 case lands at 0.91 — on the edge, but it passes 30 seeds, so it stays. A second full TF32 tap doubles the error to ~180% and fails. I tested four positions for it; all fail. So one tap.

Case D has two input-channel tiles, so I can apply TF32 to ci_t=0 only and leave ci_t=1 in pure fp32. The error then scales with how many TF32 taps hit the first tile, not the total, so the budget stretches further.

The X-side cast had a wrinkle: caching all eight batches in TF32 overflowed SBUF by 832 bytes, so I used a single 2KB shared buffer overwritten per batch, since the Tensor Engine consumes it right after each cast.

x_tf32 = nl.ndarray((nl.par_dim(TILE_CI), ROW_BLOCK, out_width),
                  dtype=nl.tfloat32, buffer=nl.sbuf)
for s in nl.static_range(8):
    x_tf32[...] = nisa.tensor_copy(bigs[s][v_c, kh + v_r, v_w + kw], dtype=nl.tfloat32)
    acc_buf[s] += nisa.nc_matmul(Wt_tf32, x_tf32)   # consumed before next overwrite

This took the mean score from 1.8361 to 1.8392.

Stuff that didn't work

  • Mixed-dtype matmul (TF32 weights, fp32 input), rejected with NCC_IBIR034. Both operands must share precision, which is why a TF32 tap means casting both sides.
  • PE tiling, splitting the 128-row stationary matrix into two 64-row tiles. Cost is bound by max(…, N_moving), so splitting the stationary dimension does nothing when the moving dimension is full-size. It cost +150µs.
  • Multi-core SPMD. kernel[nl.nc(2)](...) fails on trn1 with "SPMD grid with multi neuron cores is not supported." It's a trn2 feature.
  • Tuning instruction order. Block-order, drain-order and tap-order sweeps were all ±2µs noise, because skip_middle_end_transformations schedules matmuls from the dependency graph, not source order.
  • An 8th TF32 tap on the 5×5 case, which passed stress at 97% of budget. Technically a pass, but 97% across 30 seeds is the kind of thing that fails on seed 31, so I capped it at 7.
